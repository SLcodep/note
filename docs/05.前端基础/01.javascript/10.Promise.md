## await xxx和Promise.reslove(xxx).then()?

`await` 本质上等价于 **暂停当前函数执行 + 将后续代码放到 Promise.resolve().then() 的回调中执行**

:::tip
- **非Promise值，创建新的resolved状态promise**
**第一步**:`await xxx` 会先将 `xxx` 包装为 `Promise`（等价于 `Promise.resolve(xxx)`）
**第二步**:暂停当前 `async` 函数的执行， **将函数的后续代码（await 下面的代码）放入这个 Promise 的 .then() 回调中**
**第三步**：JS 引擎继续执行当前事件循环的同步代码，直到同步栈为空；
**第四步**：当 `Promise` 状态变为 `resolved` 时，.then() 回调（即 await 后的代码）被推入 “微任务队列”，等待下一轮事件循环执行。
```js
async function fn() {
  console.log('fn内部同步代码');
  await 123; // 等价于 Promise.resolve(123).then(后续代码)
  console.log('await后代码'); // 这行是微任务
}
fn()

//上面代码等价于
Promise.reslove(123).then(()=>console.log('await后代码'))
//而Promise.resolve(123)等价于
const p = new Promise((resolve,reject)=>{
  resolve(123)
}).then(
  ()=>console.log('await后代码')
)

```
:::

### **题目1: 翻译成Promise**
下面的问题中`await getJson()`先执行`getJson()`返回了一个`Promise`对象。
而为什么不是之前提到的`Promise.resolve(getJson()).then(后续代码)`,这是因为Promise的**透传**。只有当 `await` 后跟「非 Promise 值」时，才会翻译为 `Promise.resolve(值).then`。

:::details Promise透传
Promise 透传特指：
>对于`Promise.resolve(x)`,**当 x 本身已是 Promise 实例时，Promise.resolve(x) 不会创建新的 Promise，而是直接返回 x 本身，相当于“穿透”了包装层，这就是透传**。
:::

![](/js/await1.png)

:::tip
- **执行顺序**

`getJson()`立即执行返回一个pending状态的Promise，同时启动2秒的定时器；当执行到resolve(2)后，状态由`pending`转换为`resolved`后，.then() 注册的回调函数才会被放入微任务队列，等待执行。
:::

```js
function getJson() {
  return new Promise((resolve, reject) => {
    setTimeout(function () {
      console.log(2);
      resolve(2);
    }, 2000);
  });
}

async function testAsync() {
  return getJson().then(()=>{
    console.log(3);
  })
  
}

testAsync();
```

## async/await原理
`async/await`
- `async` 函数本质是对 `Generator` 函数的封装，JS 引擎会将 `async` 函数转换为 **Generator + 自动执行器**的组合
- `await` 关键字对应 `Generator` 中的 `yield `表达式 —— 暂停函数执行并等待 `Promise` 决议，自动执行器则负责驱动 `Generator` 函数逐步执行，最终将异步逻辑以同步写法呈现。

核心是 **用 Generator 暂停执行，用 Promise 处理异步，用自动执行器完成流程驱动**

## 代码输出:Promise.all

`Promise.all` 是 JavaScript 中处理多个并行 Promise 的核心方法，它接收一个可迭代对象（通常是数组） 作为参数，返回一个新的 Promise。

>简单比喻：你点了三杯奶茶（三个异步任务），Promise.all 会等所有奶茶都做好（所有 Promise 都成功），才把三杯一起给你；只要有一杯做砸了（一个 Promise 失败），立刻告诉你 “订单失败”，且只返回失败的原因。

```js
// 正确写法：传入由 Promise.resolve 生成的已成功 Promise 数组
const p3 = Promise.all([
  Promise.resolve(1), // 已成功的 Promise，值为1
  Promise.resolve(2), // 已成功的 Promise，值为2
  Promise.resolve(3)  // 已成功的 Promise，值为3
]);
p3.then(res => console.log(res)); // 输出：[1,2,3]

// 等价于上面的写法（JS 自动把数字转为 Promise.resolve(数字)）
const p4 = Promise.all([1, 2, 3]);
p4.then(res => console.log(res)); // 输出：[1,2,3]
```

## 代码输出:new Promise

![](/js/Promise1.png)

```js
1
finally2
finally
finally2后面的then函数 2
```

:::tip
上面代码中的一个关键点就是`.finally()`会透传上一个Promise的结果，而不会传递自己的返回值。
`.finally()` 的设计初衷是做 “清理工作”（无论成功失败都执行），所以它的行为是：
  - 接收上一个 Promise 的状态和值（成功 / 失败 + 对应值）
  - 执行自己的回调函数
  - **返回一个新的 Promise，其状态和值与上一个 Promise 完全相同**，忽略自己回调函数的返回值（除非返回一个 rejected Promise）

除此之外，`Promise.resolve('1')`是同步执行，并将后面的`then`加入到微任务队列中，再执行`Promise.resolve('2')`后将`finally`后的代码加入微任务队列，这时候开始清空微任务队列。
:::
